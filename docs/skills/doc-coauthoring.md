# 文档协作 (Doc Co-authoring) 指南

## 触发条件

当用户提及撰写文档、创建提案、起草规范或类似结构化内容时使用此技能。使用结构化工作流引导用户完成文档协作。触发条件：用户提及写文档、创建提案、起草规范或类似文档任务。

---

## 概述

本技能提供结构化工作流，引导用户完成协作文档创建。作为主动指南，引导用户完成三个阶段：上下文收集、精炼与结构、以及读者测试。

## 何时提供此工作流

**触发条件：**
- 用户提及写文档："写一个文档"、"起草提案"、"创建一个规范"、"写下来"
- 用户提及特定文档类型："PRD"、"设计文档"、"决策文档"、"RFC"
- 用户似乎正在开始实质性写作任务

**初始提供：**
向用户提供结构化文档协作工作流。解释三个阶段：

1. **上下文收集**：用户提供所有相关背景，而 Claude 提出澄清问题
2. **精炼与结构**：通过头脑风暴和编辑迭代构建每个部分
3. **读者测试**：在其他人阅读之前，用全新的 Claude（无上下文）测试文档以发现盲点

解释这种方法有助于确保文档在其他人阅读时效果良好（包括当他们将其粘贴到 Claude 中时）。询问他们是想尝试这个工作流还是更喜欢自由形式。

如果用户拒绝，自由形式工作。如果用户接受，进入阶段 1。

## 阶段 1：上下文收集

**目标：** 缩小 Claude 和用户之间的上下文差距，使以后能够进行智能指导。

### 初始问题

首先向用户询问关于文档的元问题：

1. 这是什么类型的文档？（例如技术规范、决策文档、提案）
2. 主要受众是谁？
3. 当有人阅读它时，期望的影响是什么？
4. 有要遵循的模板或特定格式吗？
5. 还有其他需要了解的约束或背景吗？

告诉他们可以用简写或任何对他们最有效的方式倾倒信息。

**如果用户提供模板或提及文档类型：**
- 询问他们是否有要共享的模板文档
- 如果他们提供指向共享文档的链接，使用适当的集成获取它
- 如果他们提供文件，阅读它

**如果用户提及编辑现有共享文档：**
- 使用适当的集成读取当前状态
- 检查没有 alt 文本的图像
- 如果存在没有 alt 文本的图像，解释说当其他人使用 Claude 理解文档时，Claude 将无法看到它们。询问他们是否要生成 alt 文本。如果是这样，请求他们将每个图像粘贴到聊天中以生成描述性 alt 文本。

### 信息倾倒

一旦初始问题得到回答，鼓励用户倾倒他们拥有的所有背景。请求信息如：
- 关于项目/问题的背景
- 相关的团队讨论或共享文档
- 为什么没有使用替代解决方案
- 组织背景（团队动态、过去的事件、政治）
- 时间线压力或约束
- 技术架构或依赖项
- 利益相关者关注点

告诉他们不要担心组织它——只是全部说出来。提供多种提供背景的方式：
- 信息倾倒自由流动
- 指向团队频道或线程以阅读
- 指向共享文档

**如果有可用的集成**（例如 Slack、Google Drive、SharePoint 或其他 MCP 服务器），提到这些可以直接拉取背景。

**如果没有检测到集成且在 Claude.ai 或 Claude 应用中：** 建议他们可以在 Claude 设置中启用连接器，以允许直接从消息应用和文档存储中拉取背景。

告诉他们一旦完成初步倾倒，将提出澄清问题。

**在上下文收集期间：**

- 如果用户提及团队频道或共享文档：
  - 如果集成可用：告知他们内容将被读取，然后使用适当的集成
  - 如果集成不可用：解释缺乏访问。建议在 Claude 设置中启用连接器，或直接粘贴相关内容

- 如果用户提及未知的实体/项目：
  - 询问是否应该搜索连接的工具以了解更多
  - 等待用户确认后再搜索

- 当用户提供上下文时，记录正在学习的内容和仍然不清楚的内容

**提出澄清问题：**

当用户表示已完成初步倾倒（或提供实质性背景）后，提出澄清问题以确保理解：

根据上下文差距生成 5-10 个编号问题。

告诉他们可以用简写回答（例如"1: 是的，2: 见 #channel，3: 不是因为向后兼容"），链接到更多文档，指向要阅读的频道，或继续信息倾倒。对他们最有效的任何方式。

**退出条件：**
当问题显示理解时——当可以询问边缘情况和权衡而不需要解释基础知识时——已收集到足够的上下文。

**过渡：**
询问他们在此阶段是否还有更多背景要提供，或者是否是时候开始起草文档了。

如果用户想添加更多，让他们。当准备好时，进入阶段 2。

## 阶段 2：精炼与结构

**目标：** 通过头脑风暴、策划和迭代精炼，逐部分构建文档。

**对用户的说明：**
解释文档将逐部分构建。对于每个部分：
1. 将提出关于包含内容的澄清问题
2. 将头脑风暴 5-20 个选项
3. 用户将指出保留/删除/合并的内容
4. 将起草该部分
5. 将通过外科编辑进行精炼

从不确定性最多的部分开始（通常是核心提案/规范），然后处理其余部分。

**部分排序：**

如果文档结构清楚：
询问他们想从哪个部分开始。

建议从不确定性最多的部分开始。对于决策文档，通常是核心提案。对于规范，通常是技术方法。摘要部分最好最后留到。

如果用户不知道他们需要哪些部分：
根据文档类型和模板，建议 3-5 个适合该文档的部分。

询问这个结构是否有效，或者是否要调整。

**一旦结构达成一致：**

创建带有所有部分占位符文本的初始文档结构。

**如果有 artifacts 访问权限：**
使用 `create_file` 创建 artifact。这为 Claude 和用户都提供了一个脚手架。

告诉他们带有所有部分标题和简要占位符文本（如"[待写]"或"[此处内容]"）的初始结构将被创建。

提供脚手架链接，并表示是时候填写每个部分了。

**如果没有 artifacts 访问权限：**
在工作目录中创建 markdown 文件。适当命名（例如 `decision-doc.md`、`technical-spec.md`）。

告诉他们带有所有部分占位符文本的初始结构将被创建。

创建带有所有部分标题和占位符文本的文件。

确认文件名已创建，并表示是时候填写每个部分了。

**对于每个部分：**

### 步骤 1：澄清问题

宣布开始处理 [部分名称] 部分。关于应该包含什么，提出 5-10 个澄清问题：

根据上下文和部分目的生成 5-10 个具体问题。

告诉他们可以用简写或只是指出什么重要。

### 步骤 2：头脑风暴

对于 [部分名称] 部分，头脑风暴可能包含的 5-20 件事，取决于部分的复杂性。寻找：
- 可能被遗忘的共享背景
- 尚未提及的角度或考虑

根据部分复杂性生成 5-10 个编号选项。最后，如果他们想要更多选项，提议再头脑风暴一些。

### 步骤 3：策划

询问哪些要点应该保留、删除或合并。请求简要理由以帮助了解后续部分的优先级。

提供示例：
- "保留 1,4,7,9"
- "删除 3（与 1 重复）"
- "删除 6（受众已经知道这个）"
- "合并 11 和 12"

**如果用户给出自由形式反馈**（例如"看起来不错"或"我大部分都喜欢，但..."）而不是编号选择，提取他们的偏好并继续。解析他们想要保留/删除/更改的内容并应用。

### 步骤 4：差距检查

根据他们选择的内容，询问 [部分名称] 部分是否有任何重要遗漏。

### 步骤 5：起草

使用 `str_replace` 将该部分的占位符文本替换为实际起草的内容。

宣布 [部分名称] 部分将根据他们的选择起草。

**如果使用 artifacts：**
起草后，提供 artifact 链接。

请他们阅读并指出要更改的内容。注意具体性有助于学习下一部分。

**如果使用文件（无 artifacts）：**
起草后，确认完成。

告诉他们 [部分名称] 部分已在 [文件名] 中起草。请他们阅读并指出要更改的内容。注意具体性有助于学习下一部分。

**对用户的 Key 指令（起草第一部分时包括）：**
提供一个提示：要求他们直接编辑文档，而是指示要更改什么。这有助于学习他们未来的风格。例如："删除 X 要点——已经被 Y 涵盖"或"使第三段更简洁"。

### 步骤 6：迭代精炼

当用户提供反馈时：
- 使用 `str_replace` 进行编辑（永远不要重新打印整个文档）
- **如果使用 artifacts：** 每次编辑后提供 artifact 链接
- **如果使用文件：** 只需确认编辑完成
- 如果用户直接编辑文档并要求阅读它：记住所做的更改并在脑海中保留，以便为后续部分记住（这显示了他们的偏好）

**继续迭代**，直到用户对该部分满意。

### 质量检查

当连续 3 次迭代没有实质性更改时，询问是否可以删除而不失去重要信息。

当部分完成时，确认 [部分名称] 完成。询问是否准备好进入下一部分。

**对所有部分重复。**

### 接近完成

当接近完成（80%+ 部分完成）时，宣布意图重新阅读整个文档并检查：
- 跨部分的流程和一致性
- 冗余或矛盾
- 任何感觉像"敷衍"或通用填充物的东西
- 每一句是否有分量

阅读整个文档并提供反馈。

**当所有部分都已起草和精炼时：**
宣布所有部分都已起草。表示意图再次审查完整文档。

审查整体连贯性、流程和完整性。

提供任何最终建议。

询问是准备进入读者测试，还是想进一步精炼任何内容。

## 阶段 3：读者测试

**目标：** 用全新的 Claude（无上下文泄漏）测试文档，验证它对读者是否有效。

**对用户的说明：**
解释现在将进行测试，看看文档对读者是否真的有效。这发现了盲点——对作者有意义但可能让其他人困惑的事情。

### 测试方法

**如果有子代理访问权限（例如在 Claude Code 中）：**

直接执行测试，无需用户参与。

### 步骤 1：预测读者问题

宣布预测读者在尝试发现此文档时可能会问什么的意图。

生成 5-10 个读者会现实问的问题。

### 步骤 2：用子代理测试

宣布这些问题将用全新的 Claude 实例测试（无此对话的上下文）。

对于每个问题，使用文档内容和问题调用子代理。

总结 Reader Claude 每个问题答对/答错的情况。

### 步骤 3：运行额外检查

宣布将执行额外检查。

调用子代理检查歧义、错误假设、矛盾。

总结发现的任何问题。

### 步骤 4：报告和修复

如果发现问题：
报告 Reader Claude 在具体问题上遇到困难。

列出具体问题。

表示修复这些差距的意图。

返回到问题部分的精炼。

---

**如果没有子代理访问权限（例如 claude.ai 网页界面）：**

用户需要手动进行测试。

### 步骤 1：预测读者问题

询问人们在尝试发现此文档时可能会问什么。他们会在 Claude.ai 中输入什么？

生成 5-10 个读者会现实问的问题。

### 步骤 2：设置测试

提供测试说明：
1. 打开新的 Claude 对话：https://claude.ai
2. 粘贴或共享文档内容（如果使用启用了连接器的共享文档平台，则提供链接）
3. 向 Reader Claude 提出的生成问题

对于每个问题，指示 Reader Claude 提供：
- 答案
- 是否任何事情模糊或不清楚
- 文档假设已经知道什么知识/上下文

检查 Reader Claude 是否给出正确答案或误解任何事情。

### 步骤 3：额外检查

也询问 Reader Claude：
- "这个文档中什么可能对读者模糊或不清楚？"
- "这个文档假设读者已经知道什么知识或上下文？"
- "是否存在任何内部矛盾或不一致？"

### 步骤 4：根据结果迭代

询问 Reader Claude 答错或遇到困难的内容。表示修复这些差距的意图。

返回到任何问题部分的精炼。

---

### 退出条件（两种方法）

当 Reader Claude 一致正确回答问题且不提出新的差距或歧义时，文档已准备好。

## 最终审查

当 Reader Claude 测试通过时：
宣布文档已通过 Reader Claude 测试。完成之前：

1. 建议他们自己最终阅读一遍——他们拥有这个文档并对其质量负责
2. 建议仔细检查任何事实、链接或技术细节
3. 询问他们验证它是否实现了他们想要的影响

询问他们是否想要一次最终审查，或者工作是否完成。

**如果用户想要最终审查，提供。否则：**

宣布文档完成。提供一些最终提示：
- 考虑在附录中链接此对话，以便读者可以看到文档是如何开发的
- 使用附录提供深度而不使主文档膨胀
- 随着从真实读者那里获得反馈更新文档

## 有效指导的技巧

**语气：**
- 直接和程序化
- 当它影响用户行为时简要解释原因
- 不要试图"推销"方法——只是执行它

**处理偏差：**
- 如果用户想跳过阶段：询问他们是否想跳过这个并自由形式工作
- 如果用户似乎沮丧：承认这比预期花费时间更长。提供更快前进的方法
- 始终给用户调整过程的空间

**上下文管理：**
- 整个过程中，如果某事缺少上下文，主动询问
- 不要让差距积累——随着出现就解决

**Artifacts 管理：**
- 使用 `create_file` 起草完整部分
- 使用 `str_replace` 进行所有编辑
- 每次更改后提供 artifact 链接
- 永远不要为头脑风暴列表使用 artifacts——那只是对话

**质量优先于速度：**
- 不要匆忙完成阶段
- 每次迭代应该有意义的改进
- 目标是创建对读者真正有效的文档
